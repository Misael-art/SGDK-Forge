Plano Teórico de Desenvolvimento de Jogo para Mega Drive (SGDK v2.11)
IA Responsável: SGDK Master Coder & Pixel Artist
Objetivo: Guiar o desenvolvimento iterativo (sprints) de um jogo completo para Mega Drive, desde a concepção até a preparação para lançamento, utilizando a SGDK v2.11.
Metodologia: Desenvolvimento Ágil baseado em Sprints, onde cada sprint foca em um conjunto específico de funcionalidades ou conteúdos, guiado por um formulário preenchido pelo usuário.
1. Estrutura de Pastas Padrão do Projeto
Uma organização clara é fundamental. Recomendo a seguinte estrutura:
MeuJogoMegaDrive/
├── src/           # Código fonte C (.c)
├── inc/           # Arquivos de cabeçalho (.h)
├── res/           # Arquivos de recursos (compilados e fontes)
│   ├── sprites/   # Arquivos PNG/BMP para sprites
│   ├── gfx/       # Arquivos PNG/BMP para tilesets e backgrounds
│   ├── sfx/       # Arquivos WAV para efeitos sonoros
│   ├── music/     # Arquivos VGM para música
│   └── resources.res # Arquivo de definição para o rescomp
├── out/           # Arquivos de saída da compilação (.o, .map, .bin, rom.md)
├── doc/           # Documentação (README.md, Design Doc)
├── tools/         # Ferramentas auxiliares (opcional)
└── makefile       # Makefile padrão da SGDK
└── compile.bat    # (Opcional) Script para compilar no Windows
Use code with caution.
2. Fases de Desenvolvimento e Sprints
Fase 1: Concepção e Planejamento (Sprint 0)
Objetivo: Definir a visão geral, escopo, mecânicas centrais e estilo do jogo.
Atividades: Brainstorming, definição do GDD (Game Design Document) inicial, escolha do estilo visual, definição dos requisitos técnicos.
Formulário: SPRINT_00_PLANNING.md
Saídas: Documento de Design (inicial) em doc/, Formulário Sprint 0 preenchido.
Estrutura: Criação da estrutura base de pastas.
Fase 2: Protótipo e Mecânicas Fundamentais (Sprints 1-N)
Objetivo: Implementar e validar as mecânicas principais do jogo (movimento, ataque, colisão básica, etc.). Criar um protótipo jogável inicial.
Atividades: Codificação das mecânicas centrais, implementação da física básica, criação de assets placeholder (sprites/tiles simples), configuração inicial do VDP e sprites.
Formulário: SPRINT_01_PROTOTYPING.md (pode haver múltiplos sprints de prototipagem)
Saídas: Código C inicial em src/ e inc/, assets placeholder em res/ (e compilados via resources.res), protótipo funcional em out/rom.bin.
Estrutura: População das pastas src, inc, res. Criação inicial do makefile e resources.res.
Fase 3: Criação e Integração de Assets (Sprints N+1 - M)
Objetivo: Produzir os assets visuais (sprites, tiles, backgrounds) e sonoros (SFX, música) definitivos e integrá-los ao jogo.
Atividades: Criação de pixel art no estilo definido, composição de música/SFX (ou seleção de bibliotecas), atualização do arquivo resources.res, substituição de placeholders no código.
Formulário: SPRINT_02_ASSETS.md (pode haver múltiplos sprints de assets)
Saídas: Arquivos PNG/BMP/WAV/VGM em res/, arquivo resources.res atualizado, código C com assets integrados, versão visualmente mais próxima do final.
Estrutura: Atualização intensiva da pasta res e do arquivo resources.res. Modificações no código em src para carregar e usar os novos assets.
Fase 4: Design de Níveis e Conteúdo (Sprints M+1 - P)
Objetivo: Construir os níveis, fases ou mundo do jogo, posicionar inimigos, itens e eventos.
Atividades: Criação de mapas de tiles (.tmx ou similar, depois convertidos), codificação da lógica de progressão, implementação de IA básica para inimigos, design de encontros e desafios.
Formulário: SPRINT_03_LEVEL_DESIGN.md (pode haver múltiplos sprints de level design)
Saídas: Mapas de tiles (arquivos de dados ou código C), código C com lógica de níveis/eventos, inimigos posicionados e funcionais no jogo.
Estrutura: Adição de arquivos de mapa em res/gfx ou res/maps, atualização do resources.res, adição de lógica de jogo em src/.
Fase 5: Polimento e Otimização (Sprints P+1 - Q)
Objetivo: Refinar a jogabilidade, adicionar efeitos visuais/sonoros, otimizar performance e corrigir bugs.
Atividades: Implementação de efeitos (parallax, partículas), ajuste fino da física e controles, adição de UI (HUD, menus), otimização do uso de VRAM e CPU, correção de bugs identificados.
Formulário: SPRINT_04_POLISH_OPTIMIZE.md
Saídas: Jogo mais estável e polido, com efeitos adicionais, performance melhorada. Código C otimizado.
Estrutura: Refinamentos em src/, inc/ e res/. Possível adição de código assembly para otimizações críticas.
Fase 6: Testes Finais e Preparação para Lançamento (Sprint Final)
Objetivo: Realizar testes extensivos, criar a tela de título, créditos e documentação final.
Atividades: Testes QA (Quality Assurance), criação da tela de título e sequência final/créditos, escrita do README final, compilação da versão final da ROM.
Formulário: SPRINT_05_FINALIZATION.md
Saídas: ROM final (.bin ou .md) em out/, README.md completo em doc/.
Estrutura: Finalização de todos os arquivos do projeto.

4. Descrição dos Arquivos Auxiliares
.c (Arquivos Fonte C): Contêm a lógica principal do jogo, implementando as mecânicas, gerenciamento de estados, etc. Organizados na pasta src/.
.h (Arquivos de Cabeçalho): Contêm declarações de funções, definições de constantes (#define), estruturas (struct), e protótipos. Usados para organizar o código e permitir a compilação separada. Organizados na pasta inc/.
resources.res (Arquivo de Definição de Recursos): Arquivo de texto lido pela ferramenta rescomp da SGDK. Ele lista todos os assets (imagens, sons, músicas, dados binários) e como eles devem ser compilados e nomeados para uso no código C. Fica na pasta res/.
.png / .bmp (Arquivos de Imagem): Formatos comuns para criar sprites, tilesets e backgrounds em pixel art. Devem seguir as limitações de cores e dimensões do Mega Drive. Ficam nas subpastas de res/ (sprites/, gfx/). rescomp converte esses arquivos para o formato nativo do Mega Drive.
.wav (Arquivos de Som): Formato comum para efeitos sonoros curtos. rescomp pode convertê-los para o formato PCM suportado pela SGDK (usando wavtoraw ou similar internamente). Ficam em res/sfx/.
.vgm (Arquivos de Música): Formato padrão para música chiptune no Mega Drive, utilizando os canais FM e PSG. Pode ser tocado diretamente pela biblioteca XGM da SGDK. Ficam em res/music/.
.res (Arquivo Binário de Recursos - Gerado): Não é editado manualmente. É o resultado da compilação do resources.res pelo rescomp. Contém os dados dos assets prontos para serem carregados na memória do Mega Drive. Geralmente fica na pasta src/ ou res/ após a compilação dos recursos.
.md (Arquivos Markdown): Usados para documentação (README.md, GDD.md, SPRINT_XX.md). Ficam na pasta doc/ ou na raiz.
makefile: Arquivo essencial para a SGDK. Define as regras de compilação do código C, linkagem e compilação dos recursos via rescomp para gerar a ROM final. Fica na raiz do projeto.
.bat / .sh (Scripts de Compilação - Opcional): Scripts simples para automatizar o processo de compilação no Windows (.bat) ou Linux/Mac (.sh), geralmente chamando o make. Ficam na raiz.
.o (Arquivos Objeto): Arquivos intermediários gerados durante a compilação do código C. Ficam na pasta out/.
.map (Arquivo de Mapa): Gerado pelo linker, mostra como a memória foi alocada para código e dados. Útil para depuração. Fica na pasta out/.
.bin / .md (ROM do Jogo): O arquivo final executável do jogo, pronto para ser carregado em um emulador ou flashcart. Fica na pasta out/.
Este plano fornece um roteiro claro e os formulários necessários para que você, como IA especialista, possa gerar os componentes de um jogo Mega Drive de forma iterativa e organizada, sprint por sprint, com base nas especificações do usuário.
